# Soroban SDK Upgrade & Migration Guide

This guide describes how to migrate smart contract projects generated by `soroban-forge` across major versions of the `soroban-sdk` and surrounding Stellar/Soroban tooling.

## Summary of Major SDK Transitions

Stellar Smart Contracts (Soroban) undergo regular protocol and SDK updates. The most notable upgrades involve transitions like:
- **Protocol 20 to Protocol 21 / 22**
- Transition from `wasm32-unknown-unknown` to `wasm32v1-none` target (requiring Rust 1.84+)
- Renaming of the command-line interface from `soroban-cli` to `stellar-cli` (invoked as `stellar`)

---

## 1. Upgrading Cargo.toml Dependencies

To migrate your project to a new SDK major version (e.g., from `20.x` to `21.x` or `22.x`):

1. Open your contract's `Cargo.toml` file.
2. Locate the `[dependencies]` and `[dev-dependencies]` blocks.
3. Update the `soroban-sdk` version:

```toml
[dependencies]
soroban-sdk = { version = "21.0.0" } # Update this to the target major version

[dev-dependencies]
soroban-sdk = { version = "21.0.0", features = ["testutils"] }
```

4. If your contract depends on other workspace members or templates (like a token library), ensure their SDK dependencies are upgraded to the exact same version to avoid compilation errors due to type mismatches.

---

## 2. Transitioning Build Target and Toolchain

### Rust Version Requirements
Newer versions of the Soroban SDK (v21.0.0+) require **Rust 1.84.0** or newer. Ensure your `rust-toolchain.toml` or `Cargo.toml` reflects this:

```toml
[package]
rust-version = "1.84"
```

Update your local Rust toolchain:
```bash
rustup update stable
```

### WASM Target Changes
Soroban now compiles contracts to the optimized `wasm32v1-none` target instead of the older `wasm32-unknown-unknown` target.

Ensure the target is installed:
```bash
rustup target add wasm32v1-none
```

Build your contract using:
```bash
stellar contract build
```

---

## 3. Adapting Contract Source Code

### Storage Layouts and TTLs
Soroban utilizes state expiration. Upgrading SDKs may introduce changes in how Storage Types and Time-To-Live (TTL) are extended.
- Ensure that `env.storage().instance().extend_ttl(...)` or `persistent().extend_ttl(...)` calls are updated to the latest parameter expectations (typically requiring threshold and limit values).
- Review all storage accesses to ensure instances are extended before expirations.

### Event Publishing
In newer SDK versions, event topics and data are published directly via:
```rust
env.events().publish(topics, data);
```
Ensure topics are defined as `Symbol` or compatible types.

---

## 4. Upgrading the Test Suite

### Environment and Mock Auths
Mocking authentication is a critical part of Soroban tests. Ensure your test environment setup in `tests/common/mod.rs` (or similar) is updated:
```rust
pub fn test_env() -> Env {
    let env = Env::default();
    env.mock_all_auths(); // Standard in v20+
    env
}
```

If you are migrating from an old protocol version, explicit mock auth structures may need to be replaced with the simpler `env.mock_all_auths()`.

---

## 5. Updating CI/CD Workflows

If you scaffolded CI/CD workflows using `soroban-forge ci-init`:
1. The actions may refer to older Rust toolchains or cargo targets.
2. Re-run `soroban-forge ci-init --force` to update the workflows to the latest presets.
3. Review `.github/workflows/build-test.yml` to verify that the compile/build step targets `wasm32v1-none` and uses the `stellar` CLI.
